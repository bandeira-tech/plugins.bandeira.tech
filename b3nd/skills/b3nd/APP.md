# APP — building a whole b3nd app

A b3nd app is **not** a service. Don't reach for endpoints, controllers, or "the API surface". An app here is three things:

1. **A URI scheme** — the language the app's data speaks.
2. **A protocol module** — programs (classify input → codes) and handlers (codes → outputs). Pure code, no I/O.
3. **A UI** — glass. Reads URIs and observes URI patterns from the same rig the rest of the world uses.

You wire it into the user's rig and you're done. There is no server to deploy. There is no database to provision. There is no client/server split at the protocol layer — one protocol module runs on both sides.

This file is the briefing for `/b3nd:new-app`. It also covers what to do when the user shows up wanting "an app" but hasn't yet figured out what shape it has.

## Before any scaffolding — the URI table

The biggest mistake an agent can make here is to start generating files. Don't. The conversation about the URI shape **is** the design. If you skip it, every file you write will need to be rewritten.

Sit with the user. Ask three questions:

1. **What does the app contain?** Not "what does it do" — what does it *hold*. Posts, sensor readings, photos, expense rows, chats, anything. Get the noun.
2. **What identifies a single one of those?** A timestamp, a slug, a content hash, a pair (`user/post`), a UUID. There is usually a natural answer; if there isn't, that's worth surfacing.
3. **What derived surfaces does the app need?** Indexes by tag, recent-first lists, relationships, search projections. These are not separate apps — they are more URI patterns the protocol owns.

Then propose 1–3 candidate URI shapes. Walk one example URI per shape:

> "Option A: `journal://entry/{ts}/{slug}/{field}` — each entry is a subtree (`title`, `body`, `tags/*`). Recent-first via `journal://index/recent/{ts}-{slug}`.
> Option B: `journal://entry/{hash}` — content-addressed. Same indexes."

The user picks. Write the chosen URI table to `apps/<app-name>/URIS.md` as a flat list of patterns with one-line descriptions. **This file is the contract** — everything else generated by this command derives from it.

Don't move on until the URI table is on disk and the user has eyeballed it.

## Reading the rig

Confirm the active rig target (`/b3nd:targets show`). Two questions to answer before you scaffold protocol code:

- **Does any current connection own the app's URI scheme?** If yes, you'll attach the protocol there. If no, the user will need to add one (`/b3nd:rig add-connection`) at the end — note this and continue.
- **Do you need new storage?** If the app's data shape suggests a backend not currently in the rig (e.g. content-addressed blobs → S3 or IPFS), surface that *now* so the rig grows alongside the app, not after.

## Programs and handlers — one per input class

For each thing the user can submit (one post, one expense, one reading), there is one program. The program classifies the message into codes. For each distinct code kind, there is one handler. The handler turns the code into outputs.

Use the dedicated commands so the patterns stay consistent:

- `/b3nd:program <protocol-dir> <name>` per input.
- `/b3nd:handler <protocol-dir> <name>` per code kind.

Resist the urge to generate them all at once. After the URI table, walk one input end-to-end: program → handler → output URIs match the table. The user sees one full path land cleanly. Then do the rest.

## The protocol module — one file, both sides

Create `apps/<app-name>/mod.ts`. It should export:

- `programs` — an array (or registry) of the programs you scaffolded.
- `handlers` — a registry keyed by code kind.
- `status()` — a small manifest of what the protocol owns and which URI patterns it claims. The rig surfaces this via `bnd status` and `b3nd_status`.
- An installer — a function that takes a `Rig` (or a connection) and registers the protocol's programs and handlers on it.

The module **must not** decide storage. It declares "I own `journal://**`"; the rig decides where `journal://**` writes go. That separation is what lets the same module run server-side, in-browser, or as an in-process consumer.

When the user reads this module a year from now, the only thing it should tell them is the *rules*. If they wonder "where does this end up?", the answer is "ask the rig".

## The UI — a b3nd on the frontend

UI default: **Vite + React + Tailwind**, matching `b3nd-web-rig`. If the user passed `--no-ui`, skip.

The UI is itself a b3nd consumer. It is not a client to a backend. Concretely:

- It builds its **own** `Rig` in-browser. The browser rig usually combines a local store (`localStorage`/`IndexedDB`, for drafts, cache, optimistic state) with an `HttpClient` or `WsClient` pointing at the user's node (for shared data).
- It imports the **same protocol module** as the server-side rig. The same `programs` and `handlers` run in the browser to classify local writes and produce outputs that go to the local store and to the remote node.
- Reads go through `rig.read(uri)`. Reactivity goes through `rig.observe(pattern)`. There is no bespoke `fetch` wrapper. If you wrote one, you've reinvented a service layer the rig already provides.

Scaffold `apps/<app-name>/web/`:

- Vite + React + Tailwind config matching the user's preferences (and the b3nd-web-rig conventions).
- Import the protocol module by relative path. Do not duplicate it.
- One example page that lists items at the protocol's main URI pattern and a form that submits via `receive`. Wire it to the browser rig.
- Run `deno check` (or the project's check) on the new files.

For a deeper UI, follow the user's lead. The baseline page is enough to prove the loop.

## Wiring the app into the rig

Either guide the user or (with confirmation) do it yourself:

1. `/b3nd:rig add-connection <name> <store-or-url> "<app-scheme>://**"` — make sure a connection owns the URI scheme.
2. In the rig file, call the protocol module's installer with that connection so its programs and handlers register on the right routes.
3. `bnd status` — confirm the protocol shows up in the rig's manifest with the right patterns.

If anything in step 3 looks wrong, fix the rig file rather than the protocol module. The protocol module is the contract; the rig is the wiring around it.

## Hand-off — teach the user the new loop

After the scaffold, the most important thing to leave the user with is a clear sense of *how to drive the app outside the UI*. Show them:

- How to write a single record from the CLI:
  ```bash
  echo '["journal://entry/2026-06-18-122100/first-day/title", "Day one"]' | bnd send -
  ```
- How to read it:
  ```bash
  bnd read 'journal://entry/2026-06-18-122100/first-day/title'
  ```
- How to subscribe:
  ```bash
  bnd observe 'journal://entry/**'
  ```

When the user sees that the UI and the CLI drive the *same* URIs through the same rig, Data Oriented Architecture clicks. That's the moment to leave them.

## Common shapes that go wrong

- **Generating a wall of files before the URI table is agreed.** Stop and back up to the URI table conversation.
- **A "service" sneaking in.** If you find yourself proposing `POST /api/journal/entries`, you have lost the thread. The URI is the endpoint. The rig is the dispatcher. There is no API tier.
- **Duplicating the protocol module for the browser.** One module, two consumers. If you copy/paste it, future protocol changes will silently diverge.
- **Routing UI state through the protocol.** "What page is the user on" is not protocol data. Keep app-local UI state in the browser, not in URIs the rig owns.
- **Premature storage choices in the protocol.** If the protocol module mentions SQLite, Postgres, or S3 by name, those references belong in the rig file, not the protocol.

## What "the app is done" looks like

- `URIS.md` is committed and accurate.
- One protocol module that imports cleanly server-side and in the browser.
- A rig connection (or several) that owns the app's URI scheme.
- A UI page that shows real data and writes real data through the rig.
- `bnd status` recognizes the protocol.
- The user can send/read/observe the app's URIs from `bnd` without touching the UI.

If all six are true, the app is shipped. The "deploy" step is "the rig is running where it needs to run", which is a rig concern, not an app concern.
