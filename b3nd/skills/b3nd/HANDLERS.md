# HANDLERS ÔÇö pure transforms

A **code handler** is a pure async transform keyed on a code string. After a program classifies a message and returns a code (see [PROGRAMS.md](./PROGRAMS.md)), the rig looks up `handlers[code]` and calls it. The handler returns the **outputs** ÔÇö the `[uri, payload]` tuples ÔÇö that the rig will dispatch through `routes.receive`.

The program decided *what kind of message this is*. The handler decides *what tuples the rig should write because of it*. That's the whole boundary.

## The contract

A handler is a pure async function. Its inputs:

- the original `[uri, payload]` the program classified
- the program's verdict ÔÇö the code, plus optional `error` message
- a `read` lookup for confirmed state, same shape the program received

Its output is `Output[]` ÔÇö an array of `[uri, payload]` tuples. Zero, one, or many. The rig dispatches them through its `receive` route; downstream observers see each URI fire.

Handlers, like programs, are pure:

- **No `fetch`, no file I/O, no clocks.** A handler does not call Stripe, write to disk, send mail, or read the wall clock. Anything effectful goes through the rig's connections (storage clients persist; transports forward) or through reactions registered on the rig (Stripe charges fire when an `*-placed` URI lands ÔÇö see [RIG.md](./RIG.md)).
- **No new timestamps.** Time-stamped URIs were generated by whoever called `receive`. If a handler needs "the time of the placement", it parses it from the URI on `out[0]` ÔÇö not from `Date.now()`.
- **No mutable module-level state.** Two parallel dispatches must not interfere.

What handlers *can* do: read confirmed state via `read`, parse URI segments to decide what derived URIs to produce, and return tuples. That's it.

**Handler emissions skip re-classification.** When a handler returns outputs, the rig persists them through `routes.receive` connections but does not re-run programs on them. They are canonical interpretations of the originating message, not new messages. Reactions (rig-level event subscribers) run *after* broadcast lands.

## From where PROGRAMS left off ÔÇö place-order

The program returned one of three codes for the place-order message:

- `ok:order-entry-placed`
- `refuse:bad-shape` / `refuse:bad-payload` / `refuse:duplicate-placement`

Each gets one handler. None of them is the same.

**Handler for `ok:order-entry-placed`** receives the original output:

```
out = ["data://orders/o/ord_42/entries/2026-06-21T14:00:00Z-placed",
       { customer: "c_99", lines: [...], total: 4200, currency: "USD" }]
```

It returns:

```
[
  out,                                                                  // the entry itself
  ["data://orders/index/by-customer/c_99/2026-06-21T14:00:00Z-ord_42",
   { order: "ord_42" }]                                                 // the customer-orders index
]
```

Two tuples. The rig persists both. Observers watching `data://orders/o/*/entries/*-placed` see the entry; observers watching `data://orders/index/by-customer/c_99/**` see the index update. Each downstream consumer reads the URI it cares about; nothing knows the others exist.

The customer id and the timestamp come from the payload and the URI respectively, both already on `out`. The handler does not generate them.

**Handlers for the refuse codes** each return `[]`. Nothing persists, no event fires, the original consumer's `receive` call comes back with the verdict (`refuse:duplicate-placement` with its `error` field) so the UI can show it. Refusal is a first-class outcome, not an exception.

## Output shape ÔÇö the tuples

An output is `[uri, payload]`. The same tuple shape that goes into the rig. Each output:

- **Must address a URI a connection in the rig owns** for `receive`. The handler decides the URI; the rig decides which connection receives it. If the URI is one no connection owns, the rig drops the output silently.
- **May reuse the input URI** (writing the message as-is ÔÇö common for the `ok` path of validate-then-persist programs).
- **May be a derived URI under the same prefix** (indexes, projections, denormalizations ÔÇö like the by-customer index above).
- **May be a URI under a different prefix entirely** (linking, fan-out ÔÇö though most protocols keep derived writes inside their own prefix to make replication and pinning clean).

Payloads carry the data the destination URI cares about. There is no envelope, no wrapper. If the protocol needs signing, content-addressing, or encryption, those happen *inside* the payload's structure (see [CANON.md](./CANON.md) when you reach that surface) ÔÇö they are not framework requirements.

## Multiple outputs per code

The place-order handler returned two tuples. That's the normal shape for any program/handler pair that produces a derived URI in addition to the canonical write. Common multi-output patterns:

| Pattern | Inputs | Outputs |
|---|---|---|
| Persist + index | one entry write | the entry + one or more index entries |
| Persist + derive | one entry write | the entry + a recomputed running total |
| Decompose | one composite payload | many individual URIs the composite resolves into |
| Tombstone | one delete-shaped write | the deletion marker + an index removal |

A handler that returns one output is fine. A handler that returns zero outputs is fine (refusal). A handler that returns many is fine. The framework imposes no cardinality.

## The restock-inventory flow

A different protocol shape. A warehouse worker posts:

```
rig.receive([
  ["data://inventory/p/SKU_007/entries/2026-06-21T14:05:00Z-received",
   { qty: 50, lot: "L-2026-118" }]
])
```

The program at `data://inventory/`:

1. Validates the URI shape (`p/<sku>/entries/<ts>-<kind>` where `<kind> Ôłł {received, sold, adjusted, reserved}`).
2. Validates the payload (`received` requires a positive integer `qty`).
3. Reads the current on-hand: `await read("data://inventory/p/SKU_007/on-hand")`.
4. Returns `ok:inventory-movement`.

The handler for `ok:inventory-movement`:

- Parses `<kind>` from the URI's last segment.
- For `received` / `adjusted-up` / `unreserved`: adds to the on-hand.
- For `sold` / `adjusted-down` / `reserved`: subtracts.
- Returns:

```
[
  out,                                                          // persist the movement entry
  ["data://inventory/p/SKU_007/on-hand", { qty: newOnHand }]    // write the derived running total
]
```

Two tuples again. The entry is the source of truth; the on-hand is the cached projection. If the projection ever goes wrong, it's recomputable by folding the entries ÔÇö which is what makes append-only history valuable in the first place.

Note the program emitted **one** code (`ok:inventory-movement`) for all four entry kinds. The handler discriminates by parsing the URI. The protocol could alternatively split into four codes (`ok:inventory-received`, `ok:inventory-sold`, ÔÇŽ) ÔÇö both shapes work. Pick the one where the handler logic stays cohesive. If the math is the same shape (add or subtract by `qty`), one handler is cleaner. If the kinds branch into materially different outputs, more codes is cleaner.

## Anti-patterns

- **Side effects in handlers.** `fetch` to Stripe, mail sends, file I/O, `Date.now()`. Effects go to rig-level reactions or to connections that own the relevant URIs. A handler returns tuples; it doesn't *do* anything.
- **Generating timestamps in handlers.** Time comes from the URI on `out[0]`, not from the clock. Two simultaneous handler calls would otherwise produce different "now" values for the same logical event.
- **Multiple handlers for the same code.** The rig dispatches by exact code string. Two registrations collide silently or replace each other; either way, the contract is broken.
- **No handler for a code the program emits.** Equally broken. Every code published by the program needs a handler; even refuse codes need a handler that returns `[]` (the framework cannot guess that "this is a refusal so nothing should persist" ÔÇö you say so by registering the handler).
- **Output URIs no connection owns.** A handler can emit any URI it wants, but the rig only persists URIs claimed by a connection in `routes.receive`. Emitting `[notmine://x, ...]` is silent dead weight. Make sure the protocol's installer documents which URI patterns it expects connections to own.
